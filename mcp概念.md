# MCP 学习报告：大模型连接外部工具的新协议

## 一、学习背景

随着大语言模型的发展，AI 已经不再只是回答问题，还开始参与写代码、查资料、分析数据、调用工具和完成任务。但是，大模型本身并不能直接访问企业数据库、文件系统、代码仓库、日历、邮件、业务系统等外部资源。

过去，如果一个 AI 应用想连接不同系统，开发者往往需要为每个系统单独开发接口。例如，连接 GitHub 要写一套集成，连接数据库又要写一套，连接企业知识库还要再写一套。这种方式开发成本高，也不利于统一管理。

MCP，也就是 Model Context Protocol，正是在这种背景下出现的。它希望为大模型应用和外部工具之间建立一个统一的连接标准。

------

## 二、MCP 是什么

MCP 的中文可以理解为“模型上下文协议”。它是一种开放协议，用来让大模型应用以标准化方式连接外部数据源和工具。

简单来说，MCP 就像是 AI 应用和外部系统之间的“通用插头”。只要一个工具或系统按照 MCP 的规则提供能力，不同的 AI 应用就可以通过 MCP 调用它，而不需要重复开发大量定制接口。

例如，一个支持 MCP 的 AI 编程助手，可以通过 MCP 连接：

- 本地文件系统；
- GitHub 仓库；
- 数据库；
- 企业文档；
- 搜索工具；
- 项目管理系统。

这样，大模型就不只是“凭已有知识回答”，而是可以在获得授权后读取真实数据、调用真实工具，并辅助用户完成更复杂的任务。

------

## 三、MCP 想解决的问题

MCP 主要解决三个问题。

第一，**解决集成碎片化问题**。
过去每个 AI 应用和每个外部系统之间都要单独适配，开发成本很高。MCP 提供统一协议后，外部系统只需要提供 MCP Server，AI 应用只需要支持 MCP Client，双方就能更容易连接。

第二，**解决大模型缺少实时上下文的问题**。
大模型训练完成后，并不天然知道用户电脑里的文件、公司最新文档、数据库里的实时数据。MCP 可以把这些外部信息作为上下文提供给模型。

第三，**解决工具调用标准不统一的问题**。
不同平台都有自己的工具调用方式。MCP 试图把“工具如何被发现、如何被调用、如何返回结果”统一起来，让工具能力可以在不同 AI 应用之间复用。

------

## 四、MCP 的基本架构

MCP 的架构可以分成三个核心角色：Host、Client 和 Server。

### 1. Host

Host 是用户直接使用的 AI 应用，比如 AI 桌面助手、AI 编程工具、企业智能助手等。它负责和用户交互，也负责管理模型、权限和上下文。

### 2. Client

Client 是 Host 内部用于连接 MCP Server 的组件。一个 Client 通常对应一个 MCP Server。它负责发送请求、接收响应，并维持连接状态。

### 3. Server

Server 是外部能力的提供者。它可以连接文件系统、数据库、代码仓库、搜索服务或业务系统，并把这些能力按照 MCP 标准暴露出来。

可以用一句话概括：

> Host 是 AI 应用，Client 是连接器，Server 是工具和数据的提供方。

------

## 五、MCP 提供的主要能力

MCP 中最重要的能力包括 Tools、Resources 和 Prompts。

### 1. Tools：工具调用

Tools 是可以被模型调用的操作。例如：

- 查询数据库；
- 搜索文档；
- 创建 GitHub Issue；
- 读取文件；
- 调用内部 API。

Tools 通常代表“可以执行的动作”。这也是 MCP 最有价值、但同时风险最高的部分。

### 2. Resources：资源读取

Resources 是外部系统提供给模型的上下文资料。例如：

- 文档内容；
- 代码文件；
- 数据库结构；
- 日志文件；
- 项目说明。

Resources 更偏向“读取信息”，让模型获得完成任务所需的背景材料。

### 3. Prompts：提示词模板

Prompts 是预设好的任务模板。例如：

- 代码审查模板；
- 日志分析模板；
- 周报生成模板；
- SQL 分析模板。

它可以帮助用户更规范地启动某类任务。

------

## 六、MCP 的工作流程

一个典型的 MCP 工作流程可以这样理解：

用户向 AI 应用提出任务，例如：“帮我分析这个项目最近失败的测试原因。”

AI 应用判断需要访问代码仓库和测试日志，于是通过 MCP Client 连接相关 MCP Server。

MCP Server 向 AI 应用说明自己能提供哪些工具，例如读取文件、查看提交记录、查询 CI 日志等。

模型根据任务选择合适工具，Host 在必要时向用户请求确认。

Server 执行工具调用，把结果返回给模型。

模型基于返回的数据继续分析，最后给出结论和建议。

这个流程说明，MCP 的作用不是替代大模型，而是让大模型可以更方便、更标准地使用外部工具和上下文。

------

## 七、MCP 与 RAG、Function Calling 的区别

学习 MCP 时，很容易把它和 RAG、Function Calling 混淆。

RAG 主要解决的是“让模型检索外部知识”的问题。它通常用于知识库问答，比如从文档中找资料，再让模型回答。

Function Calling 主要解决的是“模型如何调用函数”的问题。它更像是某个平台内部提供的一种工具调用机制。

MCP 则更偏底层和通用。它不仅可以支持工具调用，也可以提供资源、提示词、上下文边界和用户交互机制。可以理解为，MCP 是连接 AI 应用与外部系统的一套协议，而 RAG 和 Function Calling 更像是具体能力或技术模式。

------

## 八、MCP 的应用场景

MCP 最典型的应用场景是 AI 编程助手。通过 MCP，AI 可以读取代码仓库、分析文件、查看提交记录、理解项目结构，并辅助开发者定位问题。

第二类场景是企业知识库问答。企业可以把内部文档、制度、产品资料通过 MCP Server 暴露给 AI 助手，让员工更方便地查询信息。

第三类场景是数据分析。数据库可以通过 MCP 提供结构和查询能力，用户用自然语言提出问题，AI 调用工具查询数据，再生成分析结果。

第四类场景是办公自动化。例如 AI 可以连接日历、邮件、任务系统，帮助用户总结会议、生成待办、整理项目进度。

------

## 九、MCP 的风险与注意事项

MCP 虽然提高了 AI 的能力，但也带来了新的风险。

首先是权限风险。如果 MCP Server 连接了数据库、文件系统或企业系统，模型调用工具时可能接触敏感数据。因此，必须设置最小权限，不能让 AI 获得过大的访问范围。

其次是误操作风险。模型可能理解错误用户意图，如果工具允许写入、删除、发送邮件或修改配置，就可能造成实际损失。因此，高风险操作必须经过用户确认。

第三是提示注入风险。外部文档、网页、邮件中可能包含恶意指令，诱导模型泄露数据或调用不该调用的工具。所以，不能把外部内容完全当成可信指令。

因此，企业使用 MCP 时，应优先从只读场景开始，逐步开放更复杂的自动化能力。

------

## 十、学习收获

通过学习 MCP，我认为它的核心价值在于：**让大模型从单纯的文本生成工具，进一步变成能够连接外部系统、理解真实上下文、调用工具完成任务的智能应用入口。**

MCP 并不是让模型本身变得更聪明，而是让模型能够更规范地获取信息和执行操作。它的意义类似于为 AI 应用建立统一的“外设接口”。

不过，MCP 也不能被简单理解为插件系统。它涉及数据访问、工具执行、权限控制和安全审计。如果使用不当，可能会带来数据泄露、越权调用和误操作风险。

因此，我认为学习 MCP 时，不能只关注它“能做什么”，还要关注它“应该怎样安全地做”。

------

## 十一、总结

MCP 是大模型应用发展过程中的一种重要协议。它通过统一的方式连接 AI 应用与外部工具、数据和服务，降低了系统集成成本，也增强了大模型处理真实任务的能力。

从学习角度看，可以把 MCP 理解为三个关键词：

**连接**：连接模型与外部系统。
**上下文**：为模型提供真实、实时的数据。
**工具**：让模型在授权范围内执行具体操作。

未来，随着 AI Agent 和智能办公场景的发展，MCP 可能会成为大模型应用基础设施中的重要组成部分。但在实际应用中，必须重视权限、安全和审计，不能盲目开放高风险工具。
