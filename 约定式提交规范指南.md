# 约定式提交规范指南

## 1. 规范概述

约定式提交是一种基于 Git 提交信息的轻量级规范。它通过要求在提交信息中提供一组结构化的词汇，来建立代码、笔记与实验历史的清晰脉络。

**核心优势：**

- **高可读性：**  快速定位底层系统模块的修改历史与实验进度。
- **语义化关联：**  提交类型直接映射语义化版本控制（SemVer）。
- **自动化检索：**  结合工具链，可一键提取特定技术栈（如 eBPF、密码学）的所有相关文献、测试数据或学习笔记。

## 2. 核心语法结构

每次 Git 提交必须严格遵循以下格式：

Plaintext

```
<类型>[可选的作用域]: <描述>

[可选的正文，详细说明背景、设计思路或基准测试数据]

[可选的脚注，如关联的 Issue、BREAKING CHANGE 声明]
```

- **类型 (Type)：**  必填，说明提交的性质。
- **作用域 (Scope)：**  可选，说明影响的具体模块或技术栈，需用英文圆括号 `()` 括起。
- **描述 (Description)：**  必填，精简提炼核心动作，紧跟在冒号和空格之后。

## 3. 自定义提交类型 (Types) 字典

针对包含工作日志、技术沉淀和实验产出的综合性仓库，全局采用以下 10 个标准类型：

### 📁 核心内容沉淀

|**类型**|**目标文件夹**|**核心场景**|
| ------| ------| ------------------------------------------------------------------|
|​**​`log`​**|​`work log/`|​**日志追踪**：记录日常工作进度、规划安排、会议讨论或排错过程。|
|​**​`note`​**|​`work note/`|​**知识沉淀**：添加或修改技术原理调研、底层源码解析、学术论文摘要。|
|​**​`outcome`​**|​`outcomes/`|​**最终产出**：提交正式的结题报告、演示文档 (PPT)、数据图表或最终工具集。|

### 🛠️ 代码与实验开发

|**类型**|**核心场景**|
| ------| ------------------------------------------------------------------------------|
|​**​`exp`​**|​**实验原型**：编写验证性质的测试代码、探针草稿或概念验证 (PoC) 脚本。|
|​**​`perf`​**|​**性能调优**：涉及降低系统延迟、优化吞吐量、无锁化改造或指令集加速等修改。|
|​**​`fix`​**|​**错误修复**：修正笔记中的严重原理性错误，或修复实验脚本中的运行时 Bug。|
|​**​`refactor`​**|​**逻辑重构**：优化代码结构或重新梳理长篇笔记的小节目录（不改变外部行为和核心含义）。|

### ⚙️ 工程与仓库维护

|**类型**|**核心场景**|
| ------| -----------------------------------------------------------------|
|​**​`build`​**|​**构建配置**：调整编译系统（如 CMakeLists.txt、Makefile）或更新依赖库。|
|​**​`docs`​**|​**全局文档**：更新仓库根目录的 README、项目架构说明或环境搭建指南。|
|​**​`chore`​**|​**杂项事务**​：维护`.gitignore`、整理零碎文件、配置自动化工具等工程化调整。|
