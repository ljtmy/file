# 使用 GitLab 实现 PDCA 实习工作跟踪

## 一、什么是 PDCA

PDCA 是一种持续改进的管理方法（也叫"戴明环"），由四个阶段循环组成：

| 阶段 | 英文 | 含义 |
|------|------|------|
| P | Plan | 计划：明确目标，制定行动方案 |
| D | Do | 执行：按计划实施，记录过程 |
| C | Check | 检查：对比结果与目标，分析偏差 |
| A | Act | 处理：总结经验，制定改进措施 |

核心思想：**做事有计划、执行有记录、结果有对比、问题有改进**。每个循环都比上一个做得更好。

---

## 二、PDCA 与 GitLab 功能的映射

```
PDCA 阶段        GitLab 对应功能
─────────────────────────────────
P (计划)     →   创建 Issue（每个任务一个）
D (执行)     →   Commits / Merge Requests
C (检查)     →   Issue 评论 + Board 状态流转
A (处理)     →   Issue 评论总结 + 关闭 Issue
```

## 三、初始化步骤

### 第 1 步：创建 Label（标签体系）

在仓库中创建 Labels，路径：`仓库 → Issues → Labels → New Label`

需要创建 **2 组标签**：

**PDCA 状态标签（用于 Board 建列）：**

| 名称 | 颜色 | 含义 | 对应阶段 |
|------|------|------|---------|
| P-planned | #0052CC（蓝） | 已计划，待执行 | P |
| D-doing | #FF8C00（橙） | 执行中 | D |
| C-checking | #FFD700（黄） | 完成待检查 | C |
| A-done | #00875A（绿） | 已闭环 | A |

**分类标签（用于筛选和统计）：**

| 名称 | 颜色 | 含义 |
|------|------|------|
| Week-1 ~ Week-N | #428BCA（浅蓝） | 按周次分类 |
| feature | #003333（深绿） | 功能开发 |
| bugfix | #FF0000（红） | Bug 修复 |
| learning | #BFEFFF（浅蓝） | 学习任务 |
| refactor | #7B68EE（紫） | 代码重构 |
| doc | #C0C0C0（灰） | 文档相关 |
| meeting | #E9967A（橙红） | 会议/沟通 |

### 第 2 步：创建 Issue Board（PDCA 看板）

路径：`仓库 → Issues → Boards → New Board`

- Board 名称：`实习 PDCA 跟踪`
- 创建 4 列，分别绑定上面 4 个状态 Label：

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│  P-planned   │   D-doing    │  C-checking  │    A-done    │
│  (待规划)    │  (执行中)    │  (待检查)    │  (已闭环)    │
│              │              │              │              │
│  Issue #5    │  Issue #2    │  Issue #3    │  Issue #1    │
│  登录接口    │  导出功能    │  单元测试    │  环境搭建    │
│              │              │              │              │
│  Issue #6    │              │              │  Issue #0    │
│  数据迁移    │              │              │  项目初始化  │
└──────────────┴──────────────┴──────────────┴──────────────┘
     P阶段           D阶段          C阶段          A阶段
```

### 第 3 步：添加 Issue 模板

将以下目录结构放到仓库根目录：

```
仓库/
└── .gitlab/
    └── issue_templates/
        ├── task_plan.md        ← P（计划）模板
        └── weekly_review.md    ← C+A（复盘）模板
```

创建 Issue 时，从下拉菜单选择对应模板即可自动填充格式。

## 四、Issue 模板内容

### 模板 1：任务计划（task_plan.md）

用于 P 阶段，每个任务创建一个 Issue：

```markdown
## 任务描述
<!-- 做什么，一句话说清楚 -->


## 目标
<!-- 怎样算完成？写可衡量的标准 -->
- [ ]


## 计划时间
- 开始：YYYY-MM-DD
- 截止：YYYY-MM-DD
- 预计工时：X 天

## 拆解步骤
- [ ] 步骤1
- [ ] 步骤2
- [ ] 步骤3

## 相关资源
<!-- 文档链接、参考代码等 -->
```

### 模板 2：周度复盘（weekly_review.md）

用于 C+A 阶段，每周创建一个复盘 Issue：

```markdown
## P - 本周计划回顾

| 序号 | 任务 | Issue |
|------|------|-------|
| 1 |  |  |
| 2 |  |  |
| 3 |  |  |

## C - 检查

### 完成情况

| 任务 | 计划 | 实际 | 完成率 |
|------|------|------|--------|
|  |  |  |  |
|  |  |  |  |
|  |  |  |  |

### 偏差分析
-

### 本周亮点
-

### 本周不足
-

## A - 改进

### 下周改进措施
1.

### 延续到下周的任务
- [ ]
```

## 五、Issue 的标题和标签规范

### 标题格式

```
[Week-X] 任务简述
```

示例：
- `[Week-1] 搭建开发环境`
- `[Week-2] 完成用户登录接口开发`
- `[Week-2] 学习 Spring Security`
- `[Week-3] 周度复盘`

### 标签组合

每个 Issue 打上两类标签：

| 标签类型 | 数量 | 示例 |
|---------|------|------|
| PDCA 状态标签 | 1 个 | `P-planned` |
| 周次标签 | 1 个 | `Week-2` |
| 方向标签 | 1 个（可选） | `feature` |

## 六、日常使用流程

以一周为单位循环：

```
周一                    周二~周四                  周五
 │                        │                        │
 ▼                        ▼                        ▼
创建 Issue              提交代码                  Issue 写评论
(选 task_plan 模板)     commit / MR 关联 Issue    (选 weekly_review 模板)
打标签 P-planned        标签改为 D-doing          复盘、分析偏差
出在 Backlog 列         出在 In Progress 列        标签改为 C-checking
                                                    出在 Reviewing 列
                                                     │
                                                     ▼
                                                导师/自己检查
                                                     │
                                                     ▼
                                                写改进总结
                                                标签改为 A-done
                                                出在 Done 列
                                                关闭 Issue
                                                     │
                                                     ▼
                                                未完成项 → 新 Issue
                                                带入下周
```

### D（执行）阶段：关联代码

在 commit message 中引用 Issue 编号：

```bash
git commit -m "feat: 实现登录接口 #3"
```

或者通过 Merge Request 关联：

```markdown
# MR 描述中写
Closes #3

## 改动说明
- 新增 LoginController
- 新增 JwtService
```

MR 合并后会自动关闭 Issue。

### C（检查）阶段：在 Issue 中写复盘评论

```markdown
## 执行复盘

### 完成情况
- [x] 设计接口参数和返回值
- [x] 实现 Controller 层
- [x] 实现 Service 层逻辑
- [ ] 自测 → 延到下周

### 实际耗时 vs 计划
| 步骤 | 计划 | 实际 | 差异原因 |
|------|------|------|---------|
| 设计 | 0.5天 | 1天 | 对业务不熟悉 |
| 编码 | 1.5天 | 1天 | 比预期顺利 |
| 测试 | 1天 | 1.5天 | 测试用例设计经验不足 |

### 遇到的问题
1. JWT 密钥配置搞错，debug 了 2 小时

### 导师反馈
> 登录逻辑没问题，但异常处理不够完善
```

### A（处理）阶段：闭环总结

在 Issue 评论中写：

```markdown
## 闭环总结

### 做得好的
- 代码结构清晰，符合项目规范

### 需要改进的
- 开发前先通读配置文件，不要边写边查

### 延续项
- 参数校验 → 已创建 Issue #7 跟进
```

然后关闭 Issue。

## 七、每日记录方法

每天花 5 分钟，在对应 Issue 的评论区写 3 句话：

```markdown
## YYYY-MM-DD 进度

- **做了什么**：完成了 xxx 功能的开发
- **遇到什么**：xxx 报错，查了 2 小时
- **学到什么**：原来 xxx 需要 xxx 配置
```

---

## 八、Check 阶段的 4 个自问

每周复盘时问自己：

1. 目标达成了吗？（量化：完成率是多少）
2. 哪里做得好？（保留，标准化）
3. 哪里做得不好？（找到根因）
4. 和预期差在哪？（计划 vs 实际）

---

## 九、Act 阶段的要求

每次复盘必须有产出：

- 至少一条**可执行**的改进措施（不是"下次注意"）
- 未完成的任务要**明确流转**到下周，创建新 Issue 跟踪

## 十、月度汇总与实习报告

### 按周次标签筛选

在 Issues 页面，筛选 `Week-3` 标签，就能看到第三周的所有任务和完整记录。

### 月度汇总 = 4 周 PDCA 的拼接

```
Week 1: 完成 xxx → 偏差 → 改进
Week 2: 用改进后的方法做 xxx → 效果更好
Week 3: ...
Week 4: ...

→ 最终报告 = 成长轨迹
```

### Board Insights

GitLab Board 自带统计，可以查看：
- 每周完成的任务数量趋势
- 各状态的任务分布

## 十一、完整示例

假设第一周有以下工作：

**周一（P）创建 3 个 Issue：**

```
Issue #1: [Week-1] 搭建开发环境          标签: P-planned, Week-1, learning
Issue #2: [Week-1] 学习项目代码规范       标签: P-planned, Week-1, learning
Issue #3: [Week-1] 完成 Hello World 接口  标签: P-planned, Week-1, feature
```

Board 状态：
```
P-planned 列: [#1, #2, #3]
D-doing 列:  (空)
C-checking 列: (空)
A-done 列:   (空)
```

**周二~周四（D）执行并关联代码：**

```
Issue #1 标签改为 D-doing → 完成 → 改为 C-checking
Issue #2 标签改为 D-doing → 完成 → 改为 C-checking
Issue #3 标签改为 D-doing → 执行中
```

Board 状态：
```
P-planned 列: (空)
D-doing 列:  [#3]
C-checking 列: [#1, #2]
A-done 列:   (空)
```

**周五（C+A）复盘并闭环：**

```
Issue #1: 写复盘评论 → 改为 A-done → 关闭
Issue #2: 写复盘评论 → 改为 A-done → 关闭
Issue #3: 未完成 → 写原因评论 → 改为 A-done → 关闭
         → 创建 Issue #4 [Week-2] 继续完成接口 标签: P-planned, Week-2
```

Board 状态：
```
P-planned 列: [#4]
D-doing 列:  (空)
C-checking 列: (空)
A-done 列:   [#1, #2, #3]
```

创建周度复盘 Issue：

```
Issue #5: [Week-1] 周度复盘  标签: Week-1, A-done
```

## 十二、为什么适合实习跟踪

| 优势 | 说明 |
|------|------|
| 结构化 | 避免"做了很多但说不清做了什么" |
| 闭环思维 | 每件事都有始有终，不会虎头蛇尾 |
| 持续改进 | 每周都能看到自己的成长轨迹 |
| 汇报友好 | 周报/实习报告的内容直接就有了 |
| 留痕可查 | 所有记录都在 GitLab 上，导师随时可看 |
